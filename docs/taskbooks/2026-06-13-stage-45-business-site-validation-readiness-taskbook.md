# 阶段 45：业务现场验证准备与采集器下载入口

> 阶段定位：商品主类识别与 SKU 自动匹配已经能跑通样例批次，下一步进入真实业务现场验证。现场用户需要能登录系统后直接下载采集器，并在清空旧业务数据后重新上传商品、创建我打模板、配置商品识别规则。

## 本阶段目标

```text
在客户管理端提供采集器下载入口。
后端提供登录态保护的采集器 ZIP 下载接口。
Docker 后端镜像内置 collector-client 目录，保证部署后接口可用。
清理当前工作区历史业务数据前先备份数据库。
保留平台账号、租户、工作区和平台解析模板。
```

## 本轮完成

```text
新增 GET /api/v1/collector-client/download。
下载接口实时打包 collector-client 目录，排除 __pycache__、构建缓存和本地运行状态。
采集连接页面顶部新增“下载采集器”按钮。
业务机配置说明改为先下载解压，再运行 order-system-collector-ui.exe；没有 exe 时运行 collector_gui.bat。
后端 Dockerfile 增加 COPY collector-client /app/collector-client。
租户端 5173 端口改为局域网可访问，平台后台仍保持本机私有。
采集器默认 API 改为 http://127.0.0.1:5173/api/v1，现场业务机可替换为 http://服务器IP:5173/api/v1。
采集器会把 服务器IP:5173 自动补全为 http://服务器IP:5173/api/v1，并阻止继续使用后端 8000 端口。
测试覆盖采集器 ZIP 下载内容。
```

## 清库边界

```text
清理范围：当前工作区业务数据和客户配置数据。
保留范围：tenants、workspaces、users、roles、user_workspaces、waybill_modes、waybill_templates、waybill_template_fields。
清理前必须备份 /data/order-system.db。
```

## 验收

```text
客户管理员登录租户端管理页面。
进入采集连接页，点击下载采集器。
下载文件为“订单整理系统采集器.zip”。
ZIP 中包含 client.py、gui.py、collector_gui.bat；如果已构建 exe，也包含 dist/order-system-collector-ui.exe。
清库后登录账号和平台解析模板仍可用。
客户可重新上传商品/SKU、创建我打模板、配置商品识别规则，并用真实采集数据跑识别预览。
```

## 当前已知缺陷 / 后续增强

```text
我打模板识别引擎当前以“打印版式指纹 + 内置商品项解析规则”为主。
单商品和多商品已统一按商品项列表读取，能覆盖当前店铺常见的商品文字、销售属性1、销售属性2、数量和备注字段。
但模板定义能力仍不是完全可配置的字段解析引擎：
1. 商品项边界主要依赖内置规则，例如逗号分段、数量标记、尺码/数量形态和备注关键字。
2. 用户暂时不能自定义“一个商品项从哪里开始、到哪里结束”的分组规则。
3. 如果后续遇到商品和规格跨多行、数量不带常见标记、备注中混入多个商品等复杂面单，可能需要继续补充解析规则或增加可视化商品项分组配置。
4. 当前先把该问题记录为引擎已知限制，不阻塞业务现场验证；等真实业务跑稳定后，再根据失败样本集中补强。
```
