# 阶段 48：现场 Alpha 整改与发布收口任务书

> 阶段定位：当前系统已经跑通真实业务主链路，但仍处于现场验证 Alpha。阶段 48 不继续无序加功能，先把现场已暴露的问题按步骤收口，形成可交付、可回滚、可复测的一版。

## 背景

近期现场验证暴露出几类问题：

```text
1. 采集器形态反复变化，业务机上出现 bat/vbs/exe 混杂、后台启动不彻底、断线重连体验不稳定。
2. 我打 / 抖店 / 菜鸟面单解析开始覆盖真实样本，但样本规则仍依赖现场临时修补，缺少固定回归集。
3. 商品/SKU 识别已经具备冲突保护和自动匹配，但精确匹配、备注字段覆盖、同名 SKU、VAP 类商品等规则需要固化。
4. 报货 Excel 已经能导出图片、档口和多版式，但版式保存、档口分配、Excel 验收仍需要产品化。
5. 工作树修改量很大，文档状态滞后，必须先整理代码和任务边界，否则后续排查、回滚和交付风险会持续升高。
```

## 总目标

```text
把当前项目从“现场边跑边修”推进到“可控客户验证版”。
完成源码收紧、采集器产品化、样本回归、导出验收和部署运维文档。
形成明确版本边界：哪些功能进入现场验证版，哪些功能排到后续阶段。
```

## 执行原则

```text
1. 先稳定主链路，再扩展高级配置。
2. 现场问题必须沉淀成测试样本，不只修当前数据。
3. 采集器交付物只保留一个正式 exe 和一份参数说明。
4. 业务机不再放 bat/vbs/临时脚本作为正式启动方式。
5. 识别不确定时必须暴露异常，不自动猜。
6. 所有重解析、清库、远程业务机操作必须先备份或明确影响范围。
```

## 分阶段整改路线

### 48.1 源码冻结与工作树收口

目标：

```text
确认当前 E:/PROJECT 项目为现场运行源码。
冻结一版“现场 Alpha 基线”。
按功能域梳理当前未提交变更，避免继续混成一团。
```

任务：

```text
[ ] 记录当前 Docker 服务、镜像、端口、数据卷和数据库路径。
[ ] 生成当前工作树变更清单，按后端、前端、采集器、文档、存储文件分类。
[ ] 删除或忽略不应入库的临时文件、预览图片、运行状态文件。
[ ] 把新增源码文件纳入版本管理视野，确认没有“漏跟踪”的核心文件。
[ ] 更新 TASKBOOK_INDEX 和 CURRENT_PROGRESS_REPORT，让文档状态追上代码。
[ ] 形成可回滚基线，后续每轮整改只做一个主题。
```

验收：

```text
git status 中不再混杂无关运行垃圾。
任务书能解释当前每个大类改动来自哪个阶段。
当前 Docker 环境可重新构建启动。
```

### 48.2 采集器产品化整改

目标：

```text
采集器正式交付物收口为“订单系统采集器.exe”。
用户通过 token / base-url / loop 参数启动，不再用账号密码登录。
业务机断线后后台等待，服务器恢复后自动重连。
```

任务：

```text
[ ] 确认 collector-client 只保留正式入口 client.py、构建脚本、图标和文档。
[ ] 重新构建“订单系统采集器.exe”，进程名和文件名一致。
[ ] ZIP 下载包只包含 exe、参数说明、示例配置，不夹带业务现场 IP。
[ ] 参数说明写清：临时检查、保存配置、后台循环、日志路径、任务计划启动。
[ ] Windows 后台启动不出现黑框；SSH 会话关闭不影响采集器。
[ ] 断网、后端重启、token 失效、采集器重启四类场景全部测试。
[ ] 三台业务机清理旧 bat/vbs/临时 exe，并按正式方式重装。
```

验收：

```text
任务管理器中显示“订单系统采集器.exe”。
关闭 SSH 连接后采集器仍在后台运行。
服务器短暂不可用时业务机不弹未处理异常，不退出。
服务器恢复后采集器自动重新注册/心跳/上传。
```

### 48.3 面单解析与识别回归集

目标：

```text
把现场出过问题的面单样本固定成测试。
抖店、菜鸟直打、我打 printXML 的核心结构不再靠手工看页面判断。
```

任务：

```text
[ ] 建立真实样本目录，脱敏保存原始 payload / printXML / 解析期望。
[ ] 固化“备注明确规格/尺码则覆盖销售属性”的我打样本。
[ ] 固化多商品拆分样本：商品项、销售属性1、销售属性2、数量、备注。
[ ] 固化 SKU 精确优先样本：黑白不能命中黑白蓝/黑白紫，VAP 不能命中范5 VAP。
[ ] 固化商品关联 1:1 样本：关联商品必须对应数据库商品，不做模糊商品绑定。
[ ] 固化冲突样本：命中多个商品或多个 SKU 时进入异常，不自动吞。
[ ] 跑通后端解析、识别、报货导出相关测试组。
```

验收：

```text
新增/现有 pytest 覆盖现场关键样本。
同一批样本重跑后识别结果稳定。
历史重解析脚本能限定 raw_record_id 或 batch，不全库误伤。
```

### 48.4 报货 Excel 与档口流程验收

目标：

```text
报货 Excel 能按真实发货流程使用。
档口库、商品默认档口、SKU 特殊档口、档口 Sheet/文档输出闭环。
```

任务：

```text
[ ] 档口库维护入口和商品/SKU 档口继承关系完成验收。
[ ] SKU 默认继承商品档口，只有特殊 SKU 才覆盖档口。
[ ] 导出表头页支持多个命名版式方案，并能应用到导出。
[ ] 评估是否将导出版式从前端 localStorage 迁移到后端工作区配置表。
[ ] 用真实批次导出 Excel，检查商品、档口、SKU 图、销售属性2、数量。
[ ] 检查异常明细表：未识别、冲突、SKU 未命中原因能让用户定位规则。
[ ] 收集用户实际拿货流程，确认合并 Sheet / 档口 Sheet / 档口文档三种输出是否保留。
```

验收：

```text
现场用户拿导出的 Excel 能直接按档口报货。
同一个批次多次导出结果一致。
版式切换后导出列、图片尺寸、行高符合预期。
```

### 48.5 部署、迁移和发布文档

目标：

```text
把开发期 Docker 和 SQLite 自动补列，收敛为可交付部署说明。
保证断电、迁移机器、拔硬盘、重启容器后可以恢复。
```

任务：

```text
[ ] 写清当前部署目录、docker compose 项目名、端口、数据卷和备份命令。
[ ] 写清数据库备份/恢复/迁移步骤。
[ ] 梳理 database.py 自动迁移与 scripts/init_db.sql 差异。
[ ] 给每个发布版本记录：镜像、源码提交、数据库备份、采集器版本。
[ ] 增加“现场操作禁令”：不直接清库、不无备份重解析、不在 SSH 会话中跑前台采集器。
[ ] 更新 README，区分开发部署、现场部署、采集器部署。
```

验收：

```text
换一台机器按文档能启动系统。
备份数据库恢复后能登录并看到商品、SKU、规则、采集记录。
采集器下载包和服务器版本能对上。
```

## 优先级任务清单

### P0：立即处理，阻塞交付

```text
[ ] 工作树分类和文档追平。
[ ] 采集器单 exe 交付物确认。
[ ] 三台业务机清理旧采集器并按正式方式后台运行。
[ ] 现场问题样本转为 pytest 回归。
[ ] 当前真实批次导出 Excel 验收。
```

### P1：本轮现场验证内完成

```text
[ ] 导出版式方案是否迁后端的技术方案确认。
[ ] 识别规则 UI 中冲突、未识别、SKU 未命中定位能力增强。
[ ] 数据库备份、恢复、重解析脚本收口。
[ ] 任务书阶段状态更新。
```

### P2：稳定后增强

```text
[ ] 更复杂的我打商品项分组配置。
[ ] 多工作空间、多用户共享导出版式。
[ ] 采集器自动升级或版本提示。
[ ] 后端正式迁移工具替代零散补列。
```

## 本阶段不做

```text
不引入新的平台类型。
不继续扩大前端页面数量。
不做大规模 UI 重皮肤。
不把识别不确定项自动归类为已匹配。
不在业务机上保留临时脚本作为正式运维方式。
```
