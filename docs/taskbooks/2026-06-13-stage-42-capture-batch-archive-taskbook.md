# 阶段 42：已采集面单按监听批次归档读取

> 日期：2026-06-13  
> 状态：已完成本轮最小整改

## 背景

采集链路已经把 `standard_details.field_values.capture_task_id` 写入每张整理面单，业务页也能按监听任务查看本轮采集结果。但客户管理页中的“打印模板规则”和“商品识别”仍然容易把当前工作区所有历史面单混在一起读取。

这会造成两个问题：

```text
1. 用户维护模板时，不知道当前样本来自哪一轮监听。
2. 用户做商品识别规则时，历史面单和本轮面单混在一起，难以复用和排查。
```

## 本轮目标

```text
打印模板规则页读取样本前，先选择监听批次。
商品识别页读取已采集面单前，先选择监听批次。
默认选择最近一轮包含整理面单的监听批次。
批次切换后，样本列表、候选面单和候选商品行都只来自当前批次。
已保存的模板定义和商品/SKU 识别规则仍按工作区保存，可跨批次复用。
```

## 业务边界

```text
1. 不改变 capture_task、raw_capture_record、standard_detail 的数据模型。
2. 不把历史面单复制到新的批次。
3. 不删除无批次归属的旧数据，但当前管理页候选只读取选中批次。
4. 模板规则本身不绑定监听批次；批次只是样本读取范围。
5. 商品识别规则本身不绑定监听批次；批次只是候选录入范围。
```

## 页面规则

打印模板规则页：

```text
加载 capture-tasks 和 standard-details。
用 standard_detail.field_values.capture_task_id 归档到监听批次。
样本数量、样本下拉和模板分组只读取当前批次。
清空或切换批次后，不再自动选中旧样本。
```

商品识别页：

```text
加载 capture-tasks 和 standard-details。
读取已采集面单候选时，只读取当前监听批次。
我打多商品面单仍按 custom_items 拆成多个候选行。
一张面单有 2 个商品时，当前批次面单数不变，候选行数可以多于面单数。
```

## 验收标准

```text
1. 前端 tenant 构建通过。
2. 租户端 Docker 容器重建后可访问。
3. 打印模板规则页显示“监听批次”下拉，默认选最近有面单的批次。
4. 打印模板规则页样本数只统计当前批次。
5. 商品识别页显示“监听批次”下拉。
6. 商品识别页切到我打模式后，候选行只来自当前批次。
7. 多商品面单在商品识别候选中仍按商品行展开。
8. 浏览器控制台无前端错误。
```

## 后续任务

```text
1. 在批次详情页展示每个批次的模板覆盖情况和未匹配原因。
2. 商品/SKU 识别执行结果接入批次维度，便于按批次生成异常。
3. Excel 导出时增加商品/SKU 匹配结果字段。
4. 大批量批次归档查询后续可增加分页和后端筛选参数。
```
