# 面单驱动设计原则

## 核心结论

订单整理系统不是根据店铺整理订单，而是根据面单信息、面单模式和用户配置整理订单。

店铺名、淘宝店铺名、抖店店铺名等，只是面单信息或标准化明细中可能出现的普通字段。系统不能把店铺作为一级业务对象，也不能把店铺作为整理主线。

## 正确主线

```text
原始采集数据
  ↓
面单模式识别
  ↓
面单模板解析
  ↓
标准化明细
  ↓
用户配置字段角色
  ↓
鞋款简称映射
  ↓
规格 / 图片 / 档口映射
  ↓
报货汇总
  ↓
Excel 导出
```

## 字段角色由用户定义

标准化明细中可能有这些字段：

```text
面单模式
完整原文
店铺名
商品简称
规格
尺码
数量
备注
图片识别
任务ID
文档ID
任务时间
采集端ID
来源机器
来源序号
原始内容
```

但这些字段的业务角色必须由用户配置。

例如：

```text
面单模式字段 = 面单模式
来源店铺字段 = 店铺名
鞋款简称字段 = 商品简称
规格字段 = 规格
尺码字段 = 尺码
数量字段 = 数量
备注字段 = 备注
```

其中 `来源店铺字段` 是可选普通字段，只能用于展示、筛选、辅助配置，不能触发系统级店铺模型。

## 禁止事项

```text
禁止根据店铺创建系统一级对象。
禁止根据店铺来决定订单整理流程。
禁止使用 shop_id 作为核心隔离字段。
禁止把店铺名写入业务规则代码。
禁止让鞋款规则、规格规则、图片规则跟店铺强绑定。
禁止把店铺字段设计为必填字段。
```

## 正确隔离方式

不同用户维护不同规则、鞋款定义、规格映射、图片绑定和档口配置。

这些配置应按 `workspace_id` 隔离，而不是按店铺隔离。

```text
User
  ↓
Workspace
  ↓
Waybill Mode
  ↓
Rules / Mappings / Images / Stalls / Exports
```

## 后续扩展

如果某些用户希望按标准化明细中的店铺名筛选或拆分报表，可以作为导出筛选条件或报表分组条件实现。

但这仍然属于用户配置层，不改变系统的面单驱动主线。
